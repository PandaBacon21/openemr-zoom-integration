## Tech Debt / Architecture Backlog (unscheduled)

Items here are known limitations or refactors that aren't tied to a sprint yet. Promote into a sprint when the cost of *not* doing the work starts to dominate.

| ID    | Area          | Size | Status  | Priority | Story / Task                              | Notes                                                                                                                                                                                                                                                                                                                                                                                                       |
| ----- | ------------- | ---: | ------- | -------- | ----------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| TD-01 | Flask Backend |    L | Backlog | Medium   | Recurring-meeting / shared Zoom meeting support | `MeetingRecord.zoom_meeting_id` is the string PK, which forces 1 MeetingRecord per Zoom meeting and breaks if a single Zoom meeting is reused across multiple appointments/encounters (recurring telehealth, shared room links). Fix requires: (1) surrogate integer PK on `MeetingRecord` with `(zoom_meeting_id, openemr_appointment_id)` uniqueness; (2) retarget FKs from `MeetingPatient` and `ClinicalNoteRecord`; (3) add `openemr_appointment_id` (and ideally `openemr_encounter_number`) to `ClinicalNoteRecord` so retrieves can disambiguate by appointment; (4) sweep every `MeetingRecord.query.filter_by(zoom_meeting_id=...)` callsite to include appointment context. Tactical mitigation in place today: `MeetingRecord.clinical_note` `order_by="ClinicalNoteRecord.received_at.desc()"` makes the single-appointment-multiple-notes case deterministic. |
| TD-02 | OpenEMR Upstream |    M | Backlog | Low      | Submit upstream PR for RsaSha384Signer multi-client JWT fix | Our `patches/RsaSha384Signer.php` (S7-08) ships a minimal kid-extraction fix as a downstream bind mount. The upstream-quality fix would also: (1) delete `$this->headers` and `modifyHeader()` — vestigial Lcobucci `Signer` interface remnants for signing, but this class is verify-only and throws on `sign()`; (2) tighten `JsonWebKeySet::getJSONWebKey()` to reject null kid when multiple keys exist instead of silently returning the first match; (3) add PHPUnit coverage under `tests/Tests/Unit/...` for kid extraction + multi-key JWKS lookup; (4) file an OpenEMR GitHub issue calling out the multi-tenant security implication (client A could authenticate as client B if A's key happened to be first in the JWKS array). Once merged + released, drop our downstream patch and remove the bind-mount line from `docker-compose.yml`. |
| TD-03 | Database     |    M | Backlog | Medium   | Enforce strict 1:1 OpenEMR provider ↔ Zoom user mapping globally | Today `ProviderMapping` only enforces uniqueness scoped to `(zoom_account_id, openemr_provider_npi)`, so the same OpenEMR provider can be mapped to different Zoom users across different registered accounts and `allow_shared_zoom_user=true` further relaxes the per-account direction. Tighten to strict global 1:1 in both directions. TODOs: (1) drop `zoom_account_id` from the duplicate check in `_create_provider_mapping` so an active mapping anywhere blocks new mappings of the same NPI — error should name the account currently holding the mapping; (2) add a partial unique index in Alembic `UNIQUE(openemr_provider_npi) WHERE is_active = true` for defense in depth; (3) add a partial unique index `UNIQUE(zoom_user_id) WHERE is_active = true` for the other direction; (4) remove `AccountConfig.allow_shared_zoom_user` flag (currently allows many providers → one Zoom user on a single account, used for testing convenience) — column + model field + API contract + React `SettingsSection` toggle + tests; (5) add a lightweight endpoint or expand `GET /config/providers` to surface globally-claimed providers so `AccountProvidersTab` can hide them from the autocomplete; (6) update integration-reference.md ProviderMapping contract and implementation-setup-guide.md "shared Zoom user behavior" note. |
